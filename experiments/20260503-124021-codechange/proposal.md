Wrap ReplayRatioController with an outer integral compensator in ContentView heartbeat to drive observed cons/prod ratio to user target

The buggy line is in ReplayRatioController.recordSelfPlayBarrierTick (forbidden file): it subtracts perTickOverhead = currentDelaySettingMs * positionsProduced / G as the self-play wall inflation per barrier tick. Under the batched shared-evaluator architecture (workers serialize through one barrier rather than running independently in parallel), this estimate is observably mis-scaled relative to actual barrier-wall inflation, and the controller's signed equilibrium settles at a cons/prod ratio meaningfully below the user-configured target (~0.78 vs target 1.10). The forbidden-file constraint rules out fixing the call site (in BatchedMoveEvaluationSource) or the controller itself. The cleanest in-allowlist intervention is to wrap the inner controller with a slow outer integral compensator in ContentView's already-existing heartbeat tick: each tick we observe gap = userTarget - snap.currentRatio and nudge the controller's INTERNAL targetRatio (T_eff) by k * gap * dt. The user-facing parameter trainingParams.replayRatioTarget does not move; only the controller's set-point drifts to compensate for the inner bias. Gain 0.05/sec is well below the inner controller's 7-second SMA bandwidth so the two loops do not fight; T_eff is bounded to [0.5, 5.0] x userTarget to prevent runaway. State is reset on session start, on user edit of replayRatioTarget, on auto-adjust toggle, and on session stop, so the compensator always re-seeds from a known-good base. Build verified.
