## Key architectural decision (made autonomously ‚Äî please review)

**The inference path keeps returning a *scalar* value (`p_win ‚àí p_loss`); the 3-wide WDL logits are exposed only for the training-graph CE loss and the metrics.** Rationale: move selection is policy-only, and the only inference-time consumers of the value are (a) display (dashboard/candidate-probe) and (b) the trainer's fresh-baseline forward ‚Äî none of which need the full W/D/L distribution; they all want the scalar (or are happy with it). Exposing only the scalar at inference avoids rewriting every inference-readback consumer (`MPSChessPlayer`, `BatchedMoveEvaluationSource`, the arena, the candidate probe, the readback `consume` closure signatures, the inference scratch buffers). The 3-wide logits live inside `ChessNetwork` (`valueLogits` tensor), are consumed by `buildTrainingOps` (CE loss) and the metrics, and never reach the inference readback. `wdl-value-head.md` ¬ß5.1 already establishes "the policy gradient only needs the scalar"; this just extends "inference only needs the scalar" too, which is true today. **If a future feature needs the WDL distribution at inference time (e.g. resign-on-confident-loss), expose a `valueDistribution` tensor on the inference path then** ‚Äî it's a small additive change at that point.

Consequence: the *code* surface shrinks a lot from the ~20-file estimate I gave the user ‚Äî the inference consumers are untouched. Net touched: `ChessNetwork.swift`, `ChessTrainer.swift`, `ReplayBuffer.swift`, `TrainingParameters.swift` + the macro pkg, `parameters.json`, the `[STATS]`-line formatter (SessionController), the chart sample/decimation if it references the dropped field, the affected tests, and the two docs.

## Other decisions
- **`valueOutput` (scalar) drops the `tanh`.** It's now `softmax(valueLogits) ¬∑ [+1, 0, ‚àí1]` = `p_win ‚àí p_loss`, which is naturally in `[‚àí1, +1]` (difference of two probabilities). No `tanh` needed; `tanh` would be redundant and would distort the [‚àí1,1] mapping. The scalar still starts at 0 with the chosen bias init (see below), matching the old `tanh(0)=0`.
- **WDL final-FC bias init `‚âà [0, ln 6, 0]` in `[win, draw, loss]` order** (‚âà `[0, 1.7918, 0]`), so `softmax` starts at `(0.125, 0.75, 0.125)` ‚Äî the empirically-draw-heavy prior. Scalar then starts at `0.125 ‚àí 0.125 = 0`. FC2 weights: He-init like the other FC layers (kept).
- **Metrics: keep `vAbs`** (mean |`p_win ‚àí p_loss`|) ‚Äî still meaningful (mean magnitude of the value estimate) ‚Äî rather than replacing it as `wdl-value-head.md` ¬ß7 suggested. **Add** `pW`/`pD`/`pL` (mean softmax probs) and `vConf` (mean `max(pW,pD,pL)` ‚Äî the WDL saturation watch). **Drop** `vBaselineDelta`. So the metric set is: existing `vMean`/`vAbs`/`vLoss` (vLoss now CE-scale), plus new `pW`/`pD`/`pL`/`vConf`, minus `vBaselineDelta`. Lower churn than a rename. `vLoss` (the value-loss metric) is now categorical CE (~0‚Äì1.5) not MSE (~0.1‚Äì0.4) ‚Äî noted for the monitoring rubric.
- **`freshBaselineMs` (cost of the fresh-baseline forward pass) is KEPT** ‚Äî it's a perf metric, not a "drift" diagnostic. Only `vBaselineDelta` (the actual drift metric) is removed.
- **The `vBaseline` *placeholder* in the training graph STAYS.** Only the buffer's *storage* of a play-time value goes away. The advantage path is unchanged: `advantage = z ‚àí vBaseline_placeholder`, where the placeholder is fed the *fresh trainer-network forward's* scalar value each step (which is what it's fed today, modulo the play-time-stored alternative we're removing). `ReplayBuffer.sample` drops its `vBaselines:` out-param; the trainer fills the placeholder feed entirely from the fresh forward.

Decisions / deviations made autonomously (review):
1. **Did NOT remove `ReplayBuffer.vBaselineStorage` / the `vBaselines:` params on `append`/`sample`** ‚Äî deviates from ¬ß5.4 of `wdl-value-head.md` and my own earlier note here. 2. **Did NOT add `value_label_smoothing_epsilon` to the `.dcmsession` saved-config block** (`SessionCheckpointFile` / `SessionController+Checkpoint`). Reason: default 0.0 = no-op; that block is a "remember the user's tuning" convenience, not load-bearing ‚Äî and the param still persists via `TrainingParameters`'s own UserDefaults. Adding it bumps the session-file schema and touches the resume override path. Follow-up if desired.
3. **Did NOT regenerate the checked-in `./parameters.json` / `parameters.md`** ‚Äî that's Stage 3 (docs); it's a generated artifact, not build-relevant, and no test reads it directly.
4. **Kept the `vBaseline` placeholder + the whole advantage/REINFORCE path exactly as-is** (confirmed: it's fed the fresh trainer-forward `valueOutput` scalar, which is now `p_win ‚àí p_loss` instead of `tanh(...)` ‚Äî still a scalar in [‚àí1,1], so the `z ‚àí vBaseline` advantage is unchanged in shape and meaning).
5. **Used `int(1 ‚àí z)` truncation, no `graph.round`** ‚Äî z ‚àà {‚àí1,0,+1} is exact in FP32 so `1 ‚àí z ‚àà {2,1,0}` truncates identity; a `-drawPenalty` rewrite (default 0) truncates to slot 1 (`<1`) or slot 2 (`‚â•1`) which is the intended contempt behavior. 6. **`sliceTensor`** is the only API in the diff not already used elsewhere in the repo; build confirms it compiles.

Post-`/recheck` fix (build still green): added an in-graph clamp of the value-target slot index to `[0, valueHeadClasses-1]` before `oneHot` ‚Äî mirrors the policy path's `max(|legal|, 1)` defensive guard.
