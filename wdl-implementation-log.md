## Key architectural decision (made autonomously — please review)

**The inference path keeps returning a *scalar* value (`p_win − p_loss`); the 3-wide WDL logits are exposed only for the training-graph CE loss and the metrics.** Rationale: move selection is policy-only, and the only inference-time consumers of the value are (a) display (dashboard/candidate-probe) and (b) the trainer's fresh-baseline forward — none of which need the full W/D/L distribution; they all want the scalar (or are happy with it). Exposing only the scalar at inference avoids rewriting every inference-readback consumer (`MPSChessPlayer`, `BatchedMoveEvaluationSource`, the arena, the candidate probe, the readback `consume` closure signatures, the inference scratch buffers). The 3-wide logits live inside `ChessNetwork` (`valueLogits` tensor), are consumed by `buildTrainingOps` (CE loss) and the metrics, and never reach the inference readback. `wdl-value-head.md` §5.1 already establishes "the policy gradient only needs the scalar"; this just extends "inference only needs the scalar" too, which is true today. **If a future feature needs the WDL distribution at inference time (e.g. resign-on-confident-loss), expose a `valueDistribution` tensor on the inference path then** — it's a small additive change at that point.

Consequence: the *code* surface shrinks a lot from the ~20-file estimate I gave the user — the inference consumers are untouched. Net touched: `ChessNetwork.swift`, `ChessTrainer.swift`, `ReplayBuffer.swift`, `TrainingParameters.swift` + the macro pkg, `parameters.json`, the `[STATS]`-line formatter (SessionController), the chart sample/decimation if it references the dropped field, the affected tests, and the two docs.

## Other decisions
- **`valueOutput` (scalar) drops the `tanh`.** It's now `softmax(valueLogits) · [+1, 0, −1]` = `p_win − p_loss`, which is naturally in `[−1, +1]` (difference of two probabilities). No `tanh` needed; `tanh` would be redundant and would distort the [−1,1] mapping. The scalar still starts at 0 with the chosen bias init (see below), matching the old `tanh(0)=0`.
- **WDL final-FC bias init `≈ [0, ln 6, 0]` in `[win, draw, loss]` order** (≈ `[0, 1.7918, 0]`), so `softmax` starts at `(0.125, 0.75, 0.125)` — the empirically-draw-heavy prior. Scalar then starts at `0.125 − 0.125 = 0`. FC2 weights: He-init like the other FC layers (kept).
- **Metrics: keep `vAbs`** (mean |`p_win − p_loss`|) — still meaningful (mean magnitude of the value estimate) — rather than replacing it as `wdl-value-head.md` §7 suggested. **Add** `pW`/`pD`/`pL` (mean softmax probs) and `vConf` (mean `max(pW,pD,pL)` — the WDL saturation watch). **Drop** `vBaselineDelta`. So the metric set is: existing `vMean`/`vAbs`/`vLoss` (vLoss now CE-scale), plus new `pW`/`pD`/`pL`/`vConf`, minus `vBaselineDelta`. Lower churn than a rename. `vLoss` (the value-loss metric) is now categorical CE (~0–1.5) not MSE (~0.1–0.4) — noted for the monitoring rubric.
- **`freshBaselineMs` (cost of the fresh-baseline forward pass) is KEPT** — it's a perf metric, not a "drift" diagnostic. Only `vBaselineDelta` (the actual drift metric) is removed.
- **The `vBaseline` *placeholder* in the training graph STAYS.** Only the buffer's *storage* of a play-time value goes away. The advantage path is unchanged: `advantage = z − vBaseline_placeholder`, where the placeholder is fed the *fresh trainer-network forward's* scalar value each step (which is what it's fed today, modulo the play-time-stored alternative we're removing). `ReplayBuffer.sample` drops its `vBaselines:` out-param; the trainer fills the placeholder feed entirely from the fresh forward.
